# 設計判断ログ

実装中に下した主要な意思決定とその理由。
「なぜそうしたか」を残すことで、ポートフォリオ的にプロダクト思考を可視化する。

## 1. 6-band フェーダから Sea ビューへ重心移動

**当初**: Sony の Headphones Connect アプリ風の 5-band EQ フェーダ。
**問題**: 既存アプリと差別化されない。"直観的"でない。
**判断**: 6-band フェーダは Manual モードとして残しつつ、メインは生体信号駆動の **Emotional Seascape** に切替。

理由:
- Sony 既存製品と被ると新規性アピールにならない
- 「EEG で何かが変わる」体験は**抽象的な数値**より**変化する世界**の方が伝わる
- 写実的な映像 (Veo 生成) を使うことで、コードベースだけでなく**作品としての完成度**を上げる

## 2. Storm シーンを Underwater に置換予定

**当初**: Calm / Golden / Storm の 3 シーン (Russell 円の象限を埋める)
**問題**: Storm は「ネガティブ感情で嵐」というメタファーで体験が暗くなる
**判断**: Underwater (水中・魚群) に置換。Arousal で魚の数・動きが活発化する優しい表現に。

理由:
- ヒーリング系コンテンツとして一貫性が出る
- Sony が掲げる "Kando" (感動) 価値観に整合
- 嵐は技術的にも生成難 (海面の動きが大きく動画ループが切れやすい)

## 3. 動画再生に QMediaPlayer ではなく OpenCV

**当初**: PyQt5 標準の QMediaPlayer + QVideoWidget
**問題**: Windows DirectShow が Veo 生成の H.264 mp4 で `0x80040266` エラー
**判断**: OpenCV `cv2.VideoCapture` でフレーム読み込み → QImage に変換 → QPainter で描画

理由:
- システムコーデックパック (K-Lite 等) のインストール不要
- フレーム単位制御 → **真のクロスフェード (alpha blend)** が可能になる副産物
- 速度倍率 (Arousal 連動) の制御も自前で可能

トレードオフ:
- CPU 負荷は若干増 (デコードを Python 側で抱える)
- ハードウェアアクセラレーションは諦め

## 4. シーン切替に hysteresis + 最小滞留

**当初**: Valence / Arousal の閾値で素直に切替
**問題**: EEG ノイズで閾値を頻繁に跨ぎ、シーンがバタつく
**判断**:
- 入る閾値と出る閾値を別にする (Schmitt trigger)
- 一度切り替わったら 6 秒は固定
- 判定用に超スロー EMA (τ≈8s) を専用に持つ

数値は実機テストで [12s/0.62/0.35] → [6s/0.57/0.40] にチューニング。

## 5. 2 軸テーマシステム (Accent × BG)

**当初**: アクセント色 5 種のみ
**判断**: アクセント 15 色 × 背景 6 パレット = 90 通り

理由:
- ポートフォリオでスクショの**バリエーション**を作れる
- 用途別 (デモ用に明るく / 集中時用に暗く) で使い分け可
- 実装コストは ThemeManager の拡張のみで済む

## 6. 音とビジュアルを**両方とも** EEG で制御

**選択肢**:
- A. 音だけ EEG 連動 (映像は静止)
- B. 映像だけ EEG 連動 (音は再生のみ)
- C. **両方** EEG 連動 ← 採用

理由:
- 単独信号 (音か絵か) だけだと「ホントに脳波で動いてる?」が伝わりにくい
- 両方シンクロして変化することで**因果関係が体感**できる
- Sony が重視する **multimodal sensing** との親和性

## 7. Public ではなく Private repo 選択

**判断**: 提出前は Private、面接時に担当者を Collaborator 招待。

理由:
- 個人 EEG 録画データを誤ってコミットしない安全弁
- 提出タイミングを自分でコントロール
- 後日 Public 化は可能 (逆は難しい)

# 設計判断ログ

実装中に下した主要な意思決定とその理由。
「なぜそうしたか」を残すことで、ポートフォリオ的にプロダクト思考を可視化する。

## 1. 6-band フェーダから Sea ビューへ重心移動

**当初**: 一般的なヘッドホンアプリ風の 5-band EQ フェーダ。
**問題**: 既存アプリと差別化されない。"直観的"でない。
**判断**: 6-band フェーダは Manual モードとして残しつつ、メインは生体信号駆動の **Emotional Seascape** に切替。

理由:
- 既存の市販イヤホン制御アプリと被ると新規性アピールにならない
- 「EEG で何かが変わる」体験は**抽象的な数値**より**変化する世界**の方が伝わる
- 写実的な映像 (Veo 生成) を使うことで、コードベースだけでなく**作品としての完成度**を上げる

## 2. Storm シーンを Underwater に置換予定

**当初**: Calm / Golden / Storm の 3 シーン (Russell 円の象限を埋める)
**問題**: Storm は「ネガティブ感情で嵐」というメタファーで体験が暗くなる
**判断**: Underwater (水中・魚群) に置換。Arousal で魚の数・動きが活発化する優しい表現に。

理由:
- ヒーリング系コンテンツとして一貫性が出る
- ヒーリング/感動体験のコンセプトに合致
- 嵐は技術的にも生成難 (海面の動きが大きく動画ループが切れやすい)

## 3. 動画再生に QMediaPlayer ではなく OpenCV

**当初**: PyQt5 標準の QMediaPlayer + QVideoWidget
**問題**: Windows DirectShow が Veo 生成の H.264 mp4 で `0x80040266` エラー
**判断**: OpenCV `cv2.VideoCapture` でフレーム読み込み → QImage に変換 → QPainter で描画

理由:
- システムコーデックパック (K-Lite 等) のインストール不要
- フレーム単位制御 → **真のクロスフェード (alpha blend)** が可能になる副産物
- 速度倍率 (Arousal 連動) の制御も自前で可能

トレードオフ:
- CPU 負荷は若干増 (デコードを Python 側で抱える)
- ハードウェアアクセラレーションは諦め

## 4. シーン切替に hysteresis + 最小滞留

**当初**: Valence / Arousal の閾値で素直に切替
**問題**: EEG ノイズで閾値を頻繁に跨ぎ、シーンがバタつく
**判断**:
- 入る閾値と出る閾値を別にする (Schmitt trigger)
- 一度切り替わったら 6 秒は固定
- 判定用に超スロー EMA (τ≈8s) を専用に持つ

数値は実機テストで [12s/0.62/0.35] → [6s/0.57/0.40] にチューニング。

## 5. 2 軸テーマシステム (Accent × BG)

**当初**: アクセント色 5 種のみ
**判断**: アクセント 15 色 × 背景 6 パレット = 90 通り

理由:
- ポートフォリオでスクショの**バリエーション**を作れる
- 用途別 (デモ用に明るく / 集中時用に暗く) で使い分け可
- 実装コストは ThemeManager の拡張のみで済む

## 6. 音とビジュアルを**両方とも** EEG で制御

**選択肢**:
- A. 音だけ EEG 連動 (映像は静止)
- B. 映像だけ EEG 連動 (音は再生のみ)
- C. **両方** EEG 連動 ← 採用

理由:
- 単独信号 (音か絵か) だけだと「ホントに脳波で動いてる?」が伝わりにくい
- 両方シンクロして変化することで**因果関係が体感**できる
- **multimodal sensing** (複数感覚同時刺激) のコンセプトと整合

## 7. Public ではなく Private repo 選択

**判断**: 提出前は Private、面接時に担当者を Collaborator 招待。

理由:
- 個人 EEG 録画データを誤ってコミットしない安全弁
- 提出タイミングを自分でコントロール
- 後日 Public 化は可能 (逆は難しい)

## 8. 3 モード構成 (Studio / Listen / Watch)

**当初**: 単一画面に全カードを並べる
**問題**: 情報過多、目的が不明瞭
**判断**: 用途別の 3 モードに分離
- **Studio**: 技術詳細表示 (科学者ビュー)
- **Listen**: 音楽体験集中 (操作 UI)
- **Watch**: 没入型 (背景 + 最小限 HUD)

→ デモ時に「これは Studio」「これは Watch」と切り替えるだけでストーリーになる.

## 9. Watch サブビュー (Surface / Underwater / City) 採用

**判断**: Watch モード内で 3 種の背景を切替可能に
- **Surface** — Veo 生成の海面 morph 動画 (Arousal で速度変化)
- **Underwater** — HR で 3 段階 (空/サンゴ+魚/ジンベエ+マンタ)
- **City** — サイバーパンク都市 (HR シンク tint + pulse)

理由:
- 「脳波だけ」じゃなく **心拍も使う** ことで Muse の信頼度の高い信号を活かす
- HR 信号は Muse の中で最も信頼度が高いので、HR 駆動シーン切替は demo 映え◎
- City は GLSL ではなく静止画 + 動的 tint で軽量に実装

## 10. パーティクル UI (Raw EEG, 神経網オーブ)

**判断**: pyqtgraph の素朴な線グラフをやめ、custom paintEvent で**パーティクル**演出
- Raw EEG: 各 ch 60 個の粒子が波形上に出生→流れる
- Watch オーブ: Fibonacci 球面 120 点の 3D 投影 + 近接接続線 + 渦巻きカラー

理由:
- アート性と技術性を同時に伝えられる
- 「**動いてる**ことが視覚的に伝わる**」 — 静止グラフだと "本当にリアルタイム?" になる
- mockup (Midjourney 風) のテイストに寄せる

トレードオフ:
- CPU 負荷あり (粒子数 / フレームレート を調整して 20-30 fps 維持)

## 11. アセット組み込み (Veo 動画 / 画像 AI 生成画像)

**判断**: PyQt5 で完璧な 3D シェーダを書く代わりに、**AI 画像/動画生成**を活用
- 海面 / 水中 / 都市夜景 → Veo (Gemini) 動画
- 回路パターン / 楽器テクスチャ → Imagen 画像

理由:
- 同じ時間で**写実度**が圧倒的に上がる
- 提出時に「**AI と協働した実装**」というメッセージが明確になる
- メンテ不要 (静的アセット)

トレードオフ:
- 動画は Git LFS で管理 (リポジトリサイズ ~40MB)
- ライセンス確認は Veo/Imagen 利用規約に従う

## 12. スプラッシュ + Hover popup + アニメーション

**判断**: 機能ではなく**第一印象**を強化する微 UX を入れる
- 起動時スプラッシュ (1.5s)
- カード hover で flat tooltip ではなく**カスタムフローティング情報パネル**
- モード切替時に **slide + opacity** アニメ

理由:
- ポートフォリオは**最初の数秒**で勝負が決まる
- 「動く UI」 = プロダクト感
- 実装コスト低 (各 30分〜1h) で印象差大
